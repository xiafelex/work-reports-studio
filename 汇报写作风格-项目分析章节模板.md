# 汇报写作风格：项目分析章节模板

## 一、用途

本文件用于统一后续各章节的汇报写法，适用于：

- 进度计划分析
- 设计分工分析
- 采购分工分析
- 物流组织分析
- 施工与移交分析

目标不是写成内部分析笔记，而是形成一种适合汇报、复盘和对外交流的文字风格。

## 二、总体风格

后续章节统一采用以下风格：

- 先给整体判断，再给分项说明
- 先说结论，再补数据和支撑
- 语言通俗、专业、紧凑，不用生涩术语
- 能用一句话说清的，不拆成复杂推理
- 能用表格或分行列示的数据，不塞进长句子里
- 文字段落负责“判断和解释”
- 表格和分行负责“并列信息和数据支撑”

## 三、推荐结构

每个分析章节优先按下面顺序组织：

### 1. 整体情况

先用一小段话说明：

- 原计划或原设想是什么
- 当前执行后的总体变化是什么
- 主要矛盾落在哪一类问题上

这一段通常是全章节最重要的判断，不要写散。

### 2. 关键数据

涉及多个节点、多个系统、多个类别时，优先用：

- 表格
- 分行
- 并列列示

不建议把多个日期、多个百分比、多个延误天数都写进同一个长句子里。

### 3. 分类判断

当一个章节里存在多类对象时，要分开讲。例如：

- 主装置和配套系统
- 关键设备和大宗主材
- 国内采购和国外采购
- 关键路径和非关键路径

这一部分的重点不是重复数据，而是说明：

- 哪一类是主要矛盾
- 哪一类不是主要矛盾
- 哪一类需要单独解释

### 4. 复盘结论

这里要把前面的分析压成几句可直接用于汇报的话。

一句话的目标是让别人听完后能马上记住：

- 项目主要问题是什么
- 主要责任边界在哪里
- 主要风险路径在哪里
- 对新项目有什么启发

### 5. 对后续策划的启发

最后一段要落回当前项目，不停留在旧项目复盘本身。

重点回答：

- 这件事对新项目意味着什么
- 后续策划应该怎么改
- 后续控制点应该怎么设

## 四、推荐语言

推荐使用的表达：

- `总体看`
- `从原计划看`
- `从执行结果看`
- `主要体现在`
- `这说明`
- `据此判断`
- `更接近`
- `不是主要原因`
- `真正影响的是`
- `应单独列项说明`
- `对新项目的直接启发是`

推荐使用的项目语言：

- `完成时间推迟`
- `达到节点`
- `影响总体进度`
- `保留一定机动时间`
- `开始影响项目里程碑`
- `单独说明`
- `前置条件未按计划完成`
- `公用工程和配套系统`
- `系统移交`
- `场外工作`
- `外部单位配合`

## 五、不建议使用的语言

下列表达尽量少用，尤其在正式汇报里：

- `打穿关键路径`
- `收口`
- `同步闭合`
- `路径持续右移`
- `末端控制线`
- `挂在WBS下`
- `打平浮时`
- `逻辑链打断`

这些词内部讨论可以用，但汇报里容易显得生硬、抽象，不够直观。

对应建议替换为：

| 不建议用语 | 建议替换 |
| --- | --- |
| `打穿关键路径` | `开始影响项目原定里程碑` |
| `收口` | `完成`、`达到节点`、`实现移交` |
| `同步闭合` | `相关前置条件未按计划完成` |
| `路径持续右移` | `相关工作持续推迟` |
| `末端控制线` | `最终影响项目完成时间的工作` |
| `挂在WBS下` | `在计划汇总中被一并统计` |

## 六、推荐段落写法

后续章节正文建议采用这种写法：

### 模式一：先整体，后数据

先写一段判断：

`本项目原计划……。从执行情况看，……。项目最终……，主要不是由于……，而是由于……所致。`

再接表格或分行数据。

### 模式二：先数据，后判断

当数据本身很关键时，可以先列：

- 节点A：原计划、更新计划、变化
- 节点B：原计划、更新计划、变化
- 节点C：原计划、更新计划、变化

然后再压一句：

`上述变化表明，项目后期主要受……影响，而不是……。`

### 模式三：分类复盘

当一个章节里有几条主线时，可以直接写：

- `A类情况`
- `B类情况`
- `C类情况`

每类只回答三件事：

- 原来怎么安排
- 后来怎么变化
- 说明了什么问题

## 七、推荐数据呈现方式

### 1. 适合用表格的情况

- 多个节点并列
- 多个系统并列
- 多个路径并列
- 同时存在原计划、更新计划、变化值

推荐字段：

- `事项`
- `原计划`
- `更新计划/实际`
- `变化`

### 2. 适合用分行的情况

- 需要列 3 到 5 条重点结论
- 不同类别不需要精确一一对比
- 重点在归纳，不在逐条对账

例如：

- 主装置总体受控
- 公用工程推迟明显
- 外部接口工作对最终完成时间影响较大

### 3. 不建议的情况

不建议把以下内容全部塞在一个长段落里：

- 多个日期
- 多个百分比
- 多个延后天数
- 多条路径名称

这会让文字显得密、绕、难读。

## 八、推荐结论写法

章节结论建议压成这种形式：

`总体看，……不是主要原因，真正影响……的是……。因此，后续……应……，并对……单独控制。`

例如进度章节的标准句式可以是：

`总体看，主装置本体不是项目后期延期的主要原因，真正影响总体完成时间的是公用工程、配套系统、系统移交以及场外相关工作。因此，后续项目策划不能只围绕主装置安排计划，还应对公用工程、场外接口和系统移交设置单独控制点。`

## 九、后续章节统一要求

后续设计、采购、物流、施工等章节，均按以下要求写：

- 先有一段整体判断
- 再列关键数据或关键分类
- 再做复盘结论
- 最后落到对新项目的启发

不追求把所有资料都写进去，只保留：

- 最关键的数据
- 最关键的判断
- 最关键的支撑

## 十、可直接复用的章节开头模板

### 进度计划分析

`本项目原计划……。从两版计划对比看，……。总体上，……不是主要原因，真正影响项目总体进度的是……。`

### 设计分工分析

`从现有资料看，项目设计工作应按……划分。总体上，……负责……，……负责……，……负责……。`

### 采购分工分析

`结合旧项目执行情况，采购分工不宜只按甲供乙供理解，更适合按……划分。总体上，……由……主导，……由……主导。`

### 物流分析

`从现有物流数据看，常规运输时间总体可控，真正影响执行节奏的不是平均运输时间，而是……。`

### 施工与移交分析

`从项目执行情况看，施工阶段的主要矛盾不完全在主装置本体，而更多体现在……。`
