# Work Reports Studio

一个面向汇报材料、PPT 版本演进、分析底稿和项目策划文稿的工作仓库。

这个仓库的目标不是做通用知识库，也不是承载所有团队过程数据，而是把和汇报交付直接相关的内容集中管理，包括：

- 项目汇报文稿与版本演进
- PPT 成品与中间版本
- 分析型 Markdown 底稿
- 支撑汇报判断的表格、图纸、进度资料和引用材料
- 局部算法或脚本型辅助资产

## 设计原则

- 交付导向：优先围绕具体汇报和输出物组织内容
- 原文可追溯：重要结论尽量能回到原始资料或中间分析稿
- 版本可复用：保留关键版本，而不是只保留最终成品
- 分析与成品并存：既保留 PPT/文档，也保留支撑判断的 Markdown 底稿
- 独立资产分仓：维护逻辑独立的仓库不直接并入当前仓库

## 当前结构

└── outputs/            # 导出结果，默认忽略

## 内容分层

### 1. 顶层项目资料

这里主要放单次汇报直接相关的文稿、PPT、PDF、Excel 和阶段性分析文档，例如：

- `马来西亚生物柴油项目-Technip汇报-中英版.md`
- `项目分工策划-设计采购物流进度版.md`
- `4.21项目分工策划报告中英文(1)-英文修订版.pptx`

适合放这里的内容通常有两个特点：

- 和当前项目汇报强相关
- 需要直接作为交付物或交付底稿维护

### 2. `排产/`

这里放排产相关专题材料，包括研究资料、排产汇报和局部算法资产。

其中 `排产/遗传算法/` 更接近一个可独立演化的小型工作模块，保留：

- Python 源码
- 输入样表
- 计算结果样例
- 可视化产物

### 3. `排产PPT素材/`

这里放更偏素材属性的参考文档与补充资料，供汇报制作时复用。

### 4. 独立仓库边界

`排产/公司管理/` 已经是独立 Git 仓库，因此当前仓库只保留其目录入口，不把它并入当前版本历史。

这样做是为了避免：

- 不同仓库历史相互污染
- 团队资产和当前项目仓库边界不清
- 临时同步产物、认证信息和依赖目录被误提交

## 默认判断规则

### 建议提交到当前仓库

- 关键汇报稿
- PPT 主版本和重要里程碑版本
- 支撑汇报结论的分析型 Markdown
- 必要的原始表格、图纸和 PDF 资料
- 能直接支撑当前交付的脚本和源码

### 默认不提交

- `tmp/`、`outputs/` 等临时导出物
- `node_modules/`、`venv/`、`__pycache__/` 等依赖和缓存
- `.auth/`、浏览器配置、编辑器缓存
- 已独立维护的子仓库内容

## 推荐工作流

### 当你开始一轮汇报迭代

1. 先收集原始资料和参考素材
2. 用 Markdown 形成分析底稿和章节结构
3. 再产出 PPT、Word 或 PDF 成品
4. 保留关键版本，删除纯临时中间物
5. 将需要长期复用的方法和风格沉淀到更稳定的说明文档中

### 当你引入一个新专题

1. 先判断它是否服务当前仓库的汇报目标
2. 如果只是局部辅助资产，可作为子目录维护
3. 如果它会长期独立演化，优先拆成独立仓库

## 当前定位

这个仓库更适合作为：

- 汇报工作台
- 项目分析底稿仓库
- PPT 与文稿版本演进仓库
- 阶段性资料整编与复用入口

换句话说：

- 面向具体交付的材料，留在这里
- 维护逻辑独立、更新频繁的团队资产，留在独立仓库
